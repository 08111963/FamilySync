========================================================
(B) UGC COMPLIANCE: SEGNALA + BLOCCA MINIMALE MA REALE
========================================================

B0) Requisiti minimi UGC (senza over-engineering)
- “Segnala contenuto” in-app (eventi, items spesa, chores; minimo 2 tipi basta ma idealmente 3)
- “Segnala utente” in-app
- “Blocca utente” in-app (deve avere un effetto osservabile: nascondere contenuti e/o ignorare eventi realtime)
- Pannello “Moderazione” per admin di famiglia: vedere report e applicare almeno 1 azione (dismiss + delete content o remove member)

B1) Schema DB (Drizzle) — aggiungi 2 tabelle nuove + (se serve) creatorUserId ai contenuti
1) Aggiungi tabella reports:
- id uuid default random
- familyId uuid NOT NULL (FK families.id)
- reporterUserId uuid NOT NULL (FK users.id)
- targetType enum: 'calendar_event' | 'shopping_item' | 'chore' | 'user'
- targetId uuid NOT NULL (id del record target o userId se targetType='user')
- reasonCategory enum: 'spam' | 'harassment' | 'hate' | 'sexual' | 'violence' | 'other'
- reasonText text nullable (limit lato API, es. 500 chars)
- status enum: 'open' | 'actioned' | 'dismissed' (default 'open')
- createdAt timestamp default now
- updatedAt timestamp default now (auto-update in app)
Indici:
- index su (familyId, status, createdAt desc)
- index su (targetType, targetId)

2) Aggiungi tabella blocks (blocchi family-scoped):
- id uuid default random
- familyId uuid NOT NULL (FK families.id)
- blockerUserId uuid NOT NULL (FK users.id)  // chi blocca (utente corrente)
- blockedUserId uuid NOT NULL (FK users.id)  // chi è bloccato
- createdAt timestamp default now
Constraint:
- unique(familyId, blockerUserId, blockedUserId)

3) Effetto “block” richiede conoscere il creatore dei contenuti.
- Verifica se calendar_events / shopping_items / chores hanno già un campo creator/createdBy.
  - Se NON esiste, aggiungi:
    - calendar_events.createdByUserId uuid NOT NULL FK users.id
    - shopping_items.createdByUserId uuid NOT NULL FK users.id
    - chores.createdByUserId uuid NOT NULL FK users.id
  - Popolamento:
    - in creazione contenuti: set createdByUserId = userId autenticato
    - per righe esistenti: fallback (se impossibile) impostare a un valore noto (es. family admin) con script one-off oppure rendere nullable e gestire null (ma preferisci NOT NULL se greenfield).

4) Aggiorna shared/schema.ts con definizioni Drizzle + drizzle-zod schema per insert/select dove serve.
5) Esegui npm run db:push e aggiusta eventuali issue di enum (usa pgEnum in Drizzle).

B2) API Moderation (nuovo router server/routes/moderation.ts)
Aggiungi queste route (tutte JSON con error model uniforme { error: { code, message, details? } }):
- POST /api/moderation/report
  body: { familyId, targetType, targetId, reasonCategory, reasonText? }
  auth required
  verifica membership in family_members
  valida che il target esista e appartenga a familyId (per content types), oppure che user sia membro (per targetType='user')
  crea report status='open'
- GET /api/moderation/reports?familyId=...
  auth required + requireFamilyAdmin(familyId)
  ritorna lista report (open prima)
- POST /api/moderation/reports/:reportId/action
  auth required + requireFamilyAdmin(familyId del report)
  body: { action: 'dismiss' | 'delete_content' | 'remove_member', notes? }
  Implementazione MINIMA:
    - dismiss: status='dismissed'
    - delete_content: se targetType != 'user', elimina target in modo SAFE scoped su familyId
    - remove_member: se targetType == 'user', rimuovi record family_members della famiglia per quel userId
  set status='actioned' (tranne dismiss)
- POST /api/moderation/block
  body: { familyId, blockedUserId }
  auth required + membership
  no self-block
  insert idempotente (se già esiste, 200)
- POST /api/moderation/unblock
  body: { familyId, blockedUserId }
- GET /api/moderation/blocks?familyId=...
  auth required + membership
  ritorna lista utenti bloccati dall’utente corrente in quella famiglia (blockedUserId + info base user)

Validazione:
- usa Zod (preferibilmente drizzle-zod + refine) per body/params/query.
- limita reasonText (es. max 500) e impedisci stringhe vuote.

B3) Enforcement blocchi (deve essere “reale”)
- Quando restituisci UGC lists:
  - calendar events GET: escludi record dove createdByUserId è in blocks (familyId + blockerUserId=currentUserId)
  - shopping_items GET: stesso filtro
  - chores GET: stesso filtro
Implementazione: prima query blocks per blockedUserIds, poi filtro in WHERE (inArray).
- Realtime:
  - Se usi websocket rooms, includi actorUserId in payload eventi creati/modificati e nel client scarta se actorUserId è bloccato (minimo).
  - (Opzionale meglio) lato server, quando emetti, non inviare a sockets di blockerUserId contenuti di blockedUserId.

B5) Termini d’Uso (update)

OUTPUT B (ACCETTAZIONE)
- Esiste report+block end-to-end: API + DB + UI.
